# Report Templates

本檔案提供 ito-debug 三種報告 variant 的 markdown 骨架。SKILL.md 步驟 6（產出階段）會讀取本檔以擷取對應骨架。

## 標準版

root cause 已確認時的預設輸出格式。

```markdown
# Root Cause 報告：[一句話描述問題]

## Root Cause
[具體到 檔名:行號 的描述。範例：`src/hooks/useUser.ts:42` 的 useEffect dependency array 漏掉 userId，導致使用者切換時不重新拉取資料]

## 觀察證據
- [證據 1：log 片段、stack trace 對應行、test 失敗輸出]
- [證據 2：行為觀察]
- [證據 3：相關 commit / blame 結果]

## 重現步驟
1. [具體可執行的步驟]
2. ...

## 影響範圍
[列出哪些功能、flow、使用者群會碰到此 bug]

## 未釐清的點
[列出尚缺資訊、無法穩定重現的條件，或可能影響重現的因素。若無此類資訊則此段省略。]

## 建議修復方向
[文字描述修復思路。不寫 code、不寫 patch；指出該動哪個 function 的哪個行為，留給後續實作者。]

## 建議回歸測試場景
- 場景 1：[觸發條件] → [預期斷言]
- 場景 2：[觸發條件] → [預期斷言]
```

## 交接版（3 次假設失敗）

3 次 hypothesis 失敗強制 handoff 時使用，status 須標為 `blocked`。

```markdown
# 交接報告：[症狀一句話]

## 症狀
[原始症狀描述]

## 已嘗試的假設
1. [假設 1] → [驗證方式] → [被排除原因]
2. [假設 2] → [驗證方式] → [被排除原因]
3. [假設 3] → [驗證方式] → [被排除原因]

## 已排除的可能 root cause
- [已確認排除的可能 root cause]

## 已收集的證據
- [log / stack trace / 環境資訊]

## 尚未釐清的點
- [尚未釐清的點]

## 建議的下一步
1. [可能的下一步調查方向]
2. [需要補的背景資料 / 工具權限]

狀態：blocked
```

## 存 local 變形（標準版 + 過程附錄）

選項（b）使用，採「標準版」骨架並於末尾追加下列段落：

```markdown
## 已排除假設（過程附錄）
- 假設 A：[內容] → [驗證方式] → [排除原因]
- 假設 B：[內容] → [驗證方式] → [排除原因]
```
