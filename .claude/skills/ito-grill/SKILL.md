# ito-grill

## 概覽

依決策樹逐分支追問，與使用者達成共識。訪談收斂後，可視使用者需求將摘要存至 `docs/ito-temp/idea/` 

## 使用時機

- 使用者說「我想討論⋯」或「幫我釐清⋯」
- 需求模糊，需要釐清細節或驗證假設
- 需要壓力測試一個計畫或設計

**不應使用的情況：** 需要直接實作的任務不使用

## 核心流程

### 步驟 1：逐一追問

依決策樹分支逐步深挖，先處理有依賴關係的決策。若設計上疑似存在脆弱假設，主動以反例、邊界條件或失敗情境切入，進行壓力測試

**四條守則：**
1. 一回合只提出一個決策問題
2. 當前一個決策解鎖新子問題且因果明確時，先用一句話宣告依賴關係，再接續提問
3. 若問題可藉由探索 codebase 取得答案，優先探索；取得結果後以一句話向使用者複述並請求確認，避免 agent 單方面假設：
   - 單一明確查詢（例如查找特定識別符或檔名）→ 直接用 `Grep` / `Glob`
   - 範圍不確定或需要多輪搜尋 → 使用 `sub-agent`（Explore 類型），避免污染主對話 context
4. 依問題性質選擇對應格式（見「問題格式」章節）

### 步驟 2：收斂判斷

當核心決策分支已達共識，主動向使用者宣告進入收斂。並列出所有未解問題供使用者選擇，最後一個選項固定為「進入下一步」：

> 核心決策已達共識，剩餘 X 個未解問題待處理：
> - A）[未解問題 1]
> - B）[未解問題 2]
> - C）進入下一步

使用者亦可在任何時點主動說「結束」或「可以了」觸發收斂

### 步驟 3：收斂後動作

詢問使用者下一步：

> 要將摘要存至 `docs/ito-temp/idea/` 供之後參考，或另有指示？

**若選擇存檔：**
1. 依訪談主題自動命名（例如 `docs/ito-temp/idea/user-auth-refactor.md`）。
2. 摘要格式由 agent 依訪談內容決定，不拘固定結構，但須涵蓋所有達成共識的決策與未解問題。
3. 建立檔案，確認路徑後告知使用者。

## 問題格式

依問題性質挑選格式。若能列出 2–4 個互斥選項，優先採用決策型或現況確認型；僅當問題本質發散時才使用開放型。

**選項設計原則：**
- 涵蓋 2–4 個最常見情況，選項之間互斥。
- 若情境複雜，可加複合選項：`D）以上都有，但⋯`。
- 開放題取得答案後，立即以決策型或現況確認型往下深挖。

### 格式 1 — 決策型

使用者需在數個方向中做出設計或產品決策，不同選項有不同技術含義。**必須附推薦與理由。**

```
---
問題 N：[問題標題]

[一到兩句情境說明，讓使用者知道為何要問這題]

 - A）[選項一]
 - B）[選項二]
 - C）[選項三，視情況加]

我的建議：B（[選項關鍵字]），理由是：
 - 1）[理由一]
 - 2）[理由二]

你傾向哪個方向？
```

### 格式 2 — 現況確認型

了解現有系統、流程或情境；使用者自身最清楚，**不需附推薦**。

```
---
問題 N：[問題標題]

[一到兩句說明為何需要確認這個現況]

 - A）[可能情況一]
 - B）[可能情況二]
 - C）[其他情況]

你們目前是哪一種？
```

### 格式 3 — 開放型

問題本質發散（背景脈絡、使用者樣貌、動機來源），列選項反而會強行收斂。

```
---
問題 N：[問題標題]

[一到兩句情境說明，點出為何這題需要開放回答]

你的情況是？
```

## 常見合理化藉口

| 合理化藉口 | 實際情況 |
|-----------|---------|
| 「這題很簡單，不需要問」 | 跳過追問會累積隱性假設，導致後續討論方向錯誤 |
| 「使用者說可以了，不用再問」 | 收斂判斷由 agent 主導，確認所有關鍵分支已覆蓋再收斂 |
| 「一次問多題比較快」 | 多題並問會讓使用者遺漏分支，破壞決策樹完整性 |
| 「使用者剛剛都同意了，直接跳到存檔」 | 需明確發出收斂提示並取得使用者確認，避免 agent 單方面結束 |

## 警訊

- 一回合出現兩個以上問題
- 問題沒有選項，也沒有開放情境說明
- 決策型問題缺少推薦與理由
- 跳過 codebase 探索直接詢問使用者技術細節
- 使用者回答超出選項範圍未經追問即繼續下一題

## 驗證

- [ ] 所有關鍵決策分支已覆蓋，無跳過
- [ ] 每題使用正確格式（決策型 / 現況確認型 / 開放型）
- [ ] 收斂提示已發出，使用者確認進入下一步
- [ ] 若選擇存檔，`docs/ito-temp/idea/[主題].md` 已建立
- [ ] 摘要內容涵蓋所有達成共識的決策與全數未解問題

## 錯誤處理

- 若使用者在訪談中途說「先跳過這題」，記錄為未解問題，併入未解問題計數。
- 若使用者回答「我不知道」：先以反推或反例切入追問（由結果、影響或相反情境導出），嘗試引導答案；若仍無法回答，記錄為未解問題，跳過續訪，併入未解問題計數。
- 若使用者回答超出選項範圍（例如描述了未列出的情況），須根據該回答追問，釐清其技術含義或決策影響，充分理解後再繼續。
