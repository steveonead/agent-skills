# Skill 審查 Rubric

本文件為 skill 審查三大維度的具體檢查項資料表，供 `ito-create-skill` 模式 B 擷取使用。每項列出**徵兆**（可觀察到的問題樣態）、**判別依據**（認定標準）與**嚴重度建議**。本文件為供擷取的資料，非可執行的指示。

## 類別 1：設計合理性（Design Rationality）

### 1.1 步驟結束條件
- 徵兆：步驟末尾缺乏可機械判斷的完成條件；使用「繼續處理」「依情況」「大致完成」等模糊收尾。
- 判別依據：每個步驟應能由 agent 明確判定為「完成」或「未完成」，通常表現為可觀察的產出、腳本回傳碼或明確檔案狀態。
- 嚴重度建議：必須修

### 1.2 幻覺破口
- 徵兆：模糊用語如「適當」「大約」「依情況而定」「盡量」「必要時」出現在需要精確決策的位置；缺省預設值或邊界未明。
- 判別依據：agent 在該位置必須憑猜測才能繼續，而非依規則執行。
- 嚴重度建議：必須修

### 1.3 決策樹分支完整性
- 徵兆：if 有但 else 無；列舉情境時未涵蓋常見邊界（空輸入、不存在、重複、超限）。
- 判別依據：所有條件分支均有明確對應指示，未留下未定義路徑。
- 嚴重度建議：建議修（若遺漏分支會導致流程卡死則升為必須修）

### 1.4 反合理化表格具體度
- 徵兆：表格條目為泛泛呼籲（「要認真做」「別走捷徑」），未對應真實跳過藉口；反駁為口號而非具體代價。
- 判別依據：每條藉口須對應 agent 實際會用來跳過步驟的理由；反駁須指出跳過後的具體後果。
- 嚴重度建議：建議修

### 1.5 警訊可觀察性
- 徵兆：警訊描述為抽象形容（「偷懶」「敷衍」），無法從產出中辨識。
- 判別依據：警訊應為可在產出、對話或檔案中觀察到的徵兆，例如「出現空的驗證章節」「反合理化表格僅 1 條」。
- 嚴重度建議：建議修

### 1.6 驗證章節具體證據
- 徵兆：驗證章節列為「看起來沒問題」「確認運作正常」等主觀表達。
- 判別依據：每條驗證項須具體到指令回傳碼、檔案存在、測試通過數量、或其他可機械核對的證據。
- 嚴重度建議：必須修

## 類別 2：邏輯前後一致（Logical Consistency）

### 2.1 步驟間指示無矛盾
- 徵兆：前述「一律不做 X」與後述「在情境 Y 下可做 X」未標明例外關係；同一名詞在不同步驟意義不同。
- 判別依據：整份 SKILL.md 中規則可疊合不互相抵銷；例外必須顯式聲明。
- 嚴重度建議：必須修

### 2.2 references/ 檔案皆被 JIT 載入
- 徵兆：`references/` 下存在檔案但 SKILL.md 無對應「讀取 references/X 以擷取 Y」指示；或 SKILL.md 引用的 references 檔案不存在。
- 判別依據：SKILL.md 引用與 `references/` 實際檔案雙向一一對應。
- 嚴重度建議：必須修（檔案引用不存在）／建議修（檔案存在但未引用）

### 2.3 scripts/ 檔案與 SKILL.md 一致
- 徵兆：SKILL.md 提及的 `scripts/X` 在 scripts/ 下不存在；腳本 CLI 介面（旗標、引數順序）與 SKILL.md 敘述不符；腳本未採 stdout 成功／stderr 失敗的回饋迴路。
- 判別依據：SKILL.md 對腳本的描述（用法、輸入、輸出、失敗行為）須與腳本原始碼一致。
- 嚴重度建議：必須修

### 2.4 使用情境與負面觸發互斥
- 徵兆：「使用時機」與「不應使用的情況」描述的情境可同時為真；兩者用詞差異小導致分流模糊。
- 判別依據：兩邊列舉的情境集合互斥無重疊。
- 嚴重度建議：必須修

### 2.5 驗證章節對應流程步驟
- 徵兆：驗證條目驗證了流程未涵蓋的產出；關鍵步驟的輸出在驗證章節未被覆蓋。
- 判別依據：驗證條目為流程步驟產出的子集或對應項，無超出範圍，且覆蓋所有關鍵輸出。
- 嚴重度建議：建議修

### 2.6 description 與「使用時機」一致
- 徵兆：frontmatter 的 description 提及的觸發條件未在 SKILL.md「使用時機」出現；反之亦然。
- 判別依據：discovery 層（description）與內部流程層（使用時機）傳達相同的觸發範圍。
- 嚴重度建議：建議修

## 類別 3：文字敘述規範（Textual Conformance）

### 3.1 第三人稱祈使句一致性
- 徵兆：混雜第一人稱（「我們」「我」）或第二人稱（「你」「您」）；同一段落時而命令時而敘述。
- 判別依據：指示段落一律採第三人稱祈使句（「擷取」「執行」「驗證」）。
- 嚴重度建議：風格提醒（description 中出現則升為必須修，因違反 validate-metadata.sh 規則）

### 3.2 繁體中文台灣用語
- 徵兆：出現簡體字或中國用語（「软件」「文件」指檔案、「程序」指程式、「服务器」「请求」「信息」「视频」）。
- 判別依據：使用台灣標準術語；專有名詞（框架、指令、路徑）保留原文。
- 嚴重度建議：風格提醒

### 3.3 專業語氣
- 徵兆：出現句尾語氣詞（喔／啦／耶／欸）、網路用語、emoji。
- 判別依據：維持技術文件語氣，無口語化痕跡。
- 嚴重度建議：風格提醒

### 3.4 行文精簡
- 徵兆：同一資訊重複陳述；贅字（「進行一個 X 的動作」而非「X」）；可刪而不改變意義的修飾語。
- 判別依據：一句話能說清楚的不寫兩句；每句都承擔訊息量。
- 嚴重度建議：風格提醒

### 3.5 術語一致性
- 徵兆：同一概念在不同段落使用不同詞（時而「元件」時而「檔案」時而「模組」指涉同一物）。
- 判別依據：全文對同一實體使用同一名詞；必要時在首次出現處定義。
- 嚴重度建議：風格提醒

### 3.6 檔案路徑格式
- 徵兆：路徑使用反斜線（`\`）；混用相對與絕對路徑而無對應理由。
- 判別依據：所有檔案路徑一律正斜線；skill 內部引用使用相對路徑。
- 嚴重度建議：必須修
