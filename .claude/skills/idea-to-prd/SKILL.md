# Idea to PRD

## 概述

將工程師的需求透過訪談確認細節、模組設計釐清邊界，最終產出讓工程師可以直接接手的 PRD。本 skill 的核心價值在於**深化模組邊界**與**釐清結構性決策**，不碰觸具體的技術實作選型。

---

## 執行流程

> 你可以根據實際情況靈活調整流程，但每個核心步驟都要確保完成，不得跳過。

---

### 步驟 1：取得需求輸入

有三種方式進入此技能，依序判斷：

**方式 A：承接前一段 context**
若對話中已有完整的訪談紀錄或使用者直接貼上訪談摘要，提取後用步驟 2 的收斂條件做 gap analysis：所有條件皆滿足則直接進入步驟 3；有缺口則帶著現有資料進入步驟 2 補問。

**方式 B：讀入需求檔案**
若使用者提供了檔案路徑（如 `docs/ideas/xxx.md` 或任何需求文件），用 Read tool 讀取後，同樣用步驟 2 的收斂條件做 gap analysis：條件已滿足則直接進入步驟 3；有缺口則進入步驟 2 補問。

**方式 C：使用者沒有提供完整的需求**
若以上兩種方式都沒有，提示使用者先用 `roast-engineer` 收斂需求（確認核心問題與使用情境），待有基本方向後再回來啟動此技能。本 skill 不負責從零開始做需求訪談。

---

### 步驟 2：針對缺口進行 Gap-filling 訪談

> 若步驟 1 的 gap analysis 顯示七項收斂條件全部滿足，直接進入步驟 3。此步驟只處理缺口，不重跑完整訪談。

**訪談方法**：採用 `roast-engineer` 的訪談格式（決策型 / 現況確認型 / 開放型），每回合只問一個最關鍵的問題。格式細節、處理「我不知道」的策略、深度優先原則等請沿用 `roast-engineer`，此處不重複定義。

**訪談範圍**：只針對尚未滿足的收斂條件補問，不延展到已確認的區域。若某個缺口問完後衍生出新的依賴問題，接著深挖到該分支收斂，再處理下一個缺口。

#### 收斂條件（七項全滿足才進入步驟 3）

1. **問題陳述已確認**：使用者同意問題陳述正確反映他們的痛點
2. **核心使用情境有具體範例**：至少一個完整的使用者故事（角色 + 觸發情境 + 預期結果）
3. **技術邊界草稿已存在**：大致知道要新建哪些模組、修改哪些現有介面（精確劃分留給步驟 4）
4. **「範疇外」清單已草擬**：明確列出這次不做什麼，避免範疇蔓延
5. **所有功能區域皆有 User Story**：每個主要功能區域都有對應使用者故事，包含 happy path 與常見 error state；若某區域完全沒有，代表訪談尚未完整
6. **成功指標已定義**：至少一個可量測的完成標準（例如：成功率 X%、錯誤頻率降為零、API P95 < X ms）
7. **未解決問題已記錄**：訪談中出現「不確定」或「之後再說」的問題，整理成清單（問題 + 影響 + Owner + 需要在何時前拿到答案）；這些問題不能被略過，它們是後續開發最容易踩坑的地方

七項皆滿足後，先向使用者朗讀問題陳述草稿並請確認，確認後進入步驟 3。

---

### 步驟 3：探索程式碼庫並產出模組地圖

> **Greenfield 專案**：若為全新專案，仍需調查既有的技術堆疊與約定慣例（framework convention、共用元件位置），避免步驟 4 的模組設計脫離專案現實。若確實完全空白才跳過此步驟。

根據步驟 2 收斂的功能邊界，用程式碼探索工具（LSP、ast-grep、Grep、Glob）調查：

- 相關模組目前在哪裡？公開介面長什麼樣？
- 使用者提到的功能是否已有部分實作？
- 有哪些既有測試或介面可作為參照？
- 有沒有隱藏的依賴或技術債會影響這個功能？

#### 產出物：模組地圖與影響面清單

探索結束後整理成兩份清單，供步驟 4 使用。這份產出物的目的是讓模組設計有程式碼現實作為依據，而不是憑空劃邊界。

**現有模組地圖**（只列與本次功能相關的）

- `{模組路徑或名稱}`：目前職責 / 主要介面 / 既有測試覆蓋概況
- ...

**影響面清單**

- 需新建：`{模組}` — 理由
- 需修改：`{現有模組}` — 修改點 + 向後相容風險
- 需觀察：`{可能被波及但未確定的模組}` — 風險描述

若探索過程中發現與訪談結論矛盾的地方（例如使用者說「沒有這個功能」但其實已有），暫停並回到步驟 2 釐清，不要自行判斷哪邊是對的。

---

### 步驟 4：規劃模組設計與結構性決策

> **深度限制**：此步驟只處理「邊界在哪」與「模組介面長什麼樣」。不決定具體技術選型——用哪個 library、哪個 pattern、哪個 API 留給實作階段。

根據步驟 2 的訪談結論與步驟 3 的模組地圖，分四個子步驟完成設計草稿，再請使用者確認。

#### 4.1 模組設計 — 追求 Deep Module

依 John Ousterhout《A Philosophy of Software Design》定義：

- **Deep Module** = 介面遠比實作簡單
- **Shallow Module** = 介面複雜度接近實作，幾乎沒有封裝價值

**判斷是否真的 Deep 的四個工具：**

1. **Information hiding**：這個模組有沒有藏起實作細節？若呼叫者需要知道內部資料結構、狀態或流程才能正確使用，就是 Shallow。
2. **Pulling complexity downward**：模組該自己吸收複雜度，讓呼叫者簡單。若很多邊界條件與錯誤處理被丟回給呼叫者，複雜度其實沒消失，只是轉嫁出去。
3. **Define errors out of existence**：介面設計上能否從根本消除某類錯誤？（例如用型別限制取代執行期檢查、用預設值取代必填參數）能消除的錯誤不該留在介面上。
4. **Different layer, different abstraction**：相鄰層級若只是 pass-through（A 呼叫 B，B 原封不動呼叫 C），中間那層沒有抽象價值，是 Shallow。

針對每個新建 / 修改的模組逐一回答這四題。若某個模組通不過任一項，考慮重新劃分邊界；若有正當理由必須保留，明確記錄「這是 Shallow Module，但必要，理由是 ___」。

#### 4.2 結構性決策清單

模組劃分完後，針對以下議題做決策。這些決策表面是產品選擇，實際上牽動長期維護成本，屬於 PRD 層級該記錄的項目：

| 類別 | 要回答的問題 |
|------|--------------|
| 狀態擁有權 | 每個模組是 stateless 還是狀態擁有者？狀態生命週期多長（跨請求、跨 session、跨部署）？ |
| 資料邊界 | 誰能寫入這份資料？誰只能讀？多租戶隔離在哪一層強制（資料庫 / API / 前端）？ |
| 橫切關注點 | Auth、audit、rate limit、log、error reporting 分別在哪一層處理？哪些必須中心化、哪些可以分散？ |
| 同步 vs 非同步 | 模組之間是同步呼叫、事件驅動，還是批次處理？選擇理由？ |
| 錯誤傳遞策略 | 錯誤往上冒泡給誰處理？哪些應該就地消化、哪些要讓使用者看到？ |

每題回答後附上「為什麼」——這些理由會直接成為 PRD 的「實作決策」章節的核心內容。若某題答不出來，八成是訪談沒觸及，回步驟 2 補問，不要自己假設答案。

#### 4.3 呈現草稿給使用者確認

將 4.1（模組設計 + Deep / Shallow 標記）與 4.2（結構性決策）整理成草稿，請使用者確認：

- 模組劃分是否符合預期？
- Shallow Module 的存在理由使用者是否接受？
- 結構性決策是否反映產品意圖？

#### 4.4 測試情境草擬

模組邊界與結構性決策確認後，針對需要測試的模組整理具體測試情境：

- **安全關鍵模組**：必須涵蓋越權存取被拒絕（含不同角色、未登入）
- **其他模組**：涵蓋正常流程 + 邊界條件 + 錯誤情況（各至少一個）
- 情境描述格式：「在 X 條件下，預期 Y」——具體到可以直接轉成 `it('...')` 的粒度，不只是「需涵蓋 X 的情境」

> **回饋訊號**：若某個測試情境需要大量 setup 或 mock 才能跑，這往往是模組耦合過重或介面不 Deep 的訊號，回到 4.1 重新檢視，而不是硬寫測試。

---

### 步驟 5：撰寫 PRD 並存檔

依照 `references/prd-template.md` 的模版撰寫 PRD，並存入專案根目錄的 `docs/prd/` 目錄。

寫 PRD 時特別注意三件事：

- **使用者故事**按功能區域分組，涵蓋所有功能面向，每組包含正常流程與常見錯誤流程
- **實作決策**只記錄產品層級的架構選擇，不寫技術實作細節——工程師應知道「要做什麼、邊界在哪」，但仍有空間自己決定「怎麼做」：
  - **產品層級**（應記錄）：狀態擁有權、資料邊界、同步 vs 非同步、橫切關注點位置、錯誤傳遞策略、資料刷新頻率
  - **技術細節**（不記錄）：用 SQS 還是 BullMQ？polling 用哪個 hook？哪個 library 的 API？
- **補充說明**的「未解決問題」子章節，要把訪談中所有「還不確定」的問題一一填入，包括問題描述、Owner、影響說明、需要在何時前拿到答案——這是給下一個接手工程師的風險地圖

存檔步驟：

- [ ] 根據功能名稱自動生成 slug（用英文關鍵詞、`-` 連接，例如 `ad-performance-view`，避免中文字元造成 CI / URL 相容問題）
- [ ] 若專案根目錄的 `docs/prd/` 不存在，先執行 `mkdir -p docs/prd`
- [ ] 以 Write tool 將 PRD 寫入 `docs/prd/{slug}.md`
- [ ] 提示使用者 PRD 路徑，以及後續可交給哪些工具繼續處理（如拆 task、開始實作）

## 常見藉口與反駁

| 藉口 | 反駁 |
|------|------|
| 「需求很清楚了，不需要訪談。」 | 「清楚」通常只是假象——gap analysis 幾乎每次都會翻出新的邊界情況。 |
| 「我先動手，之後再補 PRD。」 | 事後補寫的 PRD 幾乎都在合理化既有選擇，而不是真正的規格文件。 |
| 「我可以一次問多個問題，節省來回次數。」 | 多問同時出現會讓使用者選擇性略過難題。一次一題才能逼出真正的思考。 |
| 「使用者說不知道，就跳過這個問題吧。」 | 不知道是探索的起點，不是終點。先給範例、再試反面切入，最後才記為未解決。 |
| 「模組看起來都很單純，不用跑 Deep Module 四問。」 | 「看起來單純」常常是 Shallow Module 的偽裝。四問跑一遍只要幾分鐘，但可以避免把複雜度轉嫁給呼叫者。 |
| 「結構性決策這題使用者沒提過，先略過。」 | 沒提過 = 訪談沒碰到，不是不重要。退回步驟 2 補問，別自己假設答案。 |

## 警示紅旗

出現以下跡象時，應立即暫停並採取對應行動：

- **使用者的需求在訪談過程中一直改變，且沒有收斂的跡象** → 問題定義本身可能有誤，退回去重新確認「這個功能要解決什麼問題」
- **模組設計超過 7 個** → 建議拆分為多個 PRD，先與使用者確認功能優先順序
- **多個 Shallow Module 且沒有充分理由** → 模組邊界可能劃錯了，回 4.1 檢視是否可合併或下推複雜度
- **結構性決策清單有題目答不出來** → 訪談沒觸及，回步驟 2 補問，不要自行假設
- **PRD 已經包含過多的實作細節或待確認的技術問題** → 退回步驟 5，將技術細節移除或標記為工程師自行決定
- **使用者說「先這樣就好，之後再說」** → 追問清楚：這個問題在上線後會影響什麼？不確定的地方先記入未解決問題
- **訪談結果和程式碼探索的發現有明顯矛盾** → 暫停步驟 3，將矛盾點列出來請使用者確認哪個描述是正確的
- **使用者的回答一直是「都可以」、「看情況」** → 換具體情境範例重新切入，例如「如果是 A 客戶在 B 情況下，他會選哪個？」

## 驗證要求

完成此技能前，必須確認以下所有項目：

- [ ] 步驟 2 的七項收斂條件全部滿足，使用者已確認問題陳述
- [ ] 步驟 3 的模組地圖與影響面清單已產出（Greenfield 例外）
- [ ] 步驟 4.1 的每個模組皆已跑過 Deep Module 四問，Shallow Module 皆有明確理由
- [ ] 步驟 4.2 的結構性決策清單五大類均已回答，每題有「為什麼」
- [ ] 步驟 4.3 使用者已確認模組設計符合預期
- [ ] PRD 已包含所有步驟中討論過的決策
- [ ] `docs/prd/{slug}.md` 已成功建立
