## 成功指標

描述如何判斷這個功能「做成了」。好的指標是可量測的，例如：

- 使用者完成某操作的成功率達到 X%
- 特定錯誤發生頻率降為零
- 某 API 回應時間 < X ms（P95）

避免寫「使用者滿意度提升」這類無法客觀驗證的描述。

## 實作決策

列出訪談過程中確認的**產品層級**架構決策。每一項決策都要附取捨理由——這是給接手工程師判斷未來變更是否違反原意的依據。

**模組設計**
- 要新建或修改的模組及其職責
- 模組間依賴關係
- Deep / Shallow 標記（Shallow Module 必須附保留理由）

**結構性決策**
- 狀態擁有權：哪些模組持有狀態、狀態生命週期
- 資料邊界與多租戶隔離策略（例如：在 API 層強制過濾，前端無法繞過）
- 橫切關注點位置（Auth、audit、rate limit、log 在哪一層處理）
- 同步 vs 非同步行為選擇（例如：匯出使用同步下載，不走非同步佇列）
- 錯誤傳遞策略（哪些就地消化、哪些冒泡給使用者）
- 資料刷新策略（例如：每小時更新、60 秒 polling）
