# Roast Engineer

## 概述

當工程師面對模糊的需求時，透過一連串的訪談協助他們釐清需求細節，最後存成高密度的訪談紀錄，供 PRD 工具繼續展開規格。

## 執行流程

### 步驟 1：逐回合訪談

每回合只問**一個目前最關鍵的問題**，根據問題性質選擇格式：

**決策型問題** — 使用者需要在幾個方向中做出設計或產品決策，且不同選項有不同的技術含義，工程師可能沒意識到取捨：

```
---
問題 N：[問題標題]

[一到兩句情境說明，讓使用者知道為何要問這題]

 - A）[選項一]
 - B）[選項二]
 - C）[選項三，視情況加]

我的建議：B（[選項關鍵字]），理由是：
 - 1）[理由一]
 - 2）[理由二]

你傾向哪個方向？
```

**現況確認型問題** — 了解現有的系統、流程或情境，使用者自己最清楚，不需要建議：

```
---
問題 N：[問題標題]

[一到兩句說明為何需要確認這個現況]

 - A）[可能情況一]
 - B）[可能情況二]
 - C）[其他情況]

你們目前是哪一種？
```

**開放型問題** — 問題本質發散（例如背景脈絡、使用者樣貌、動機來源），列選項反而會強行收斂，此時不給選項：

```
---
問題 N：[問題標題]

[一到兩句情境說明，點出為何這題需要開放回答而不是從選項中選]

你的情況是？
```

> 由 agent 自行判斷何時採開放式；只要問題能列出 2–4 個互斥選項，仍優先用決策型或現況確認型。開放題拿到答案後，常會延伸出具體的決策分支，請接著往下深挖。

**選項設計原則：**
- 涵蓋最常見的 2–4 種情況，選項之間互斥
- 若情況較複雜，可加「D）以上都有，但…」複合選項

**決策樹原則（深度優先，不廣度跳躍）：**
1. **識別衍生分支**：使用者回答後，若開了新的子問題，立刻接著問，把這條分支走到底，再回下一個主題
2. **明確標示依賴**：若後續問題取決於剛才的決定，先說出來（例如：「因為你選了 B，接下來需要確認…」）
3. **目標是毫不放水**：不斷問下去，直到每條分支都有明確答案或標記為「未解決問題」

**處理「我不知道」的順序：**
1. 提供具體情境範例，讓使用者用直覺判斷
2. 若仍無法判斷，從反面切入（例如：「那你確定哪些情況**不需要**隔離？」）
3. 若兩種方式都無效，無條件記錄到「未解決問題」，繼續下一個分支

> 每個策略只試一次，無效即進下一步，不得反覆用同一個切角纏問。

> 若某個問題使用者無法回答、且答案可從 Codebase 確認（例如「現有 API 是否已支援此功能」），可用另一個 Agent 探索；若環境無 subagent，請使用者協助查證或記為未解問題。不得主動翻閱程式碼作為預設行為。

**範例對話（內部儀表板）：**

```
使用者：我想做一個內部儀表板，顯示團隊的工作進度。

agent：
---
問題 1：儀表板的主要使用者是誰？

這會決定資訊顆粒度與權限邊界。

 - A）個人貢獻者（看自己和直屬團隊）
 - B）Team lead（看整個團隊）
 - C）高層（看跨團隊趨勢）

你們目前是哪一種？

使用者：主要是 team lead。

agent：
---
問題 2：team lead 最痛的是什麼？（因為你選了 B，接著確認價值來源）

釐清這題能避免做出資訊很多但沒人看的面板。

 - A）不知道誰卡住
 - B）進度估算不準
 - C）跨專案優先順序難排

我的建議：A，理由是：
 - 1）「誰卡住」是行動觸發點，解了會立刻被使用
 - 2）估算與優先順序需要信任歷史資料，做了也可能沒人信

你傾向哪個方向？

使用者：A，但我們連「卡住」都沒有統一定義。

agent：
---
問題 3：你們團隊目前怎麼判斷一個任務「卡住」？

這題需要你先描述現況，再據此設計可偵測的訊號。

你的情況是？
...
```

這段示範了**深度優先**（問題 3 順著「卡住沒定義」這條新分支繼續深挖，而不是跳回問題 2 的其他面向）、**依賴標示**（問題 2 明說「因為你選了 B」），以及**開放型問題**（問題 3 對發散題不強行列選項）。

### 步驟 2：收斂判斷

**每回合訪談結束後**，agent 應主動自我評估一次是否達到收斂條件，不應等待使用者提出。

當目前掌握的資訊已經足以讓另一個工程師草擬 PRD，且與當前使用者的理解已達成共識時，便可以考慮收斂。此時先做一次**隱含依賴掃描**，確認核心功能的配套需求都已涵蓋，沒有明顯空白，再詢問使用者是否同意收斂。

**隱含依賴掃描**：逐一回顧已確認的核心功能，思考每個功能要「正確運作」，還需要哪些尚未討論到的資料、配套行為或邊界條件。若發現空白，補問一題，再重新評估。

判斷時可自問：
- 需求的「為什麼」清楚嗎？工程師知道優先順序的依據是什麼嗎？
- 成功標準有可判斷的依據，還是停留在空泛描述？**至少一個指標要可量測**（例如：成功率 X%、錯誤頻率降為零、API P95 < X ms）
- 核心功能邊界明確嗎（哪些在 scope、哪些不在）？
- 有沒有表面是產品選擇、但實際牽動資料模型或系統邊界的決策尚未觸及？
- 每個核心功能區域是否有至少一個具體 User Story（角色 + 觸發情境 + 預期結果）？若某個功能區域完全沒有，代表訪談尚未完整。
- 技術邊界草稿是否存在，大致知道要新建哪些模組、或修改哪些現有介面？不需要精確，但不能完全空白。
- 使用者還有沒有說不清楚的假設或疑問？

若判斷資訊已足夠，主動提示：
> 「訪談已接近收斂，目前還有 X 個未解問題待確認。要繼續深挖某條分支，還是可以進入下一步？」

### 步驟 3：存檔

使用者確認收斂後，詢問：「是否存檔？」

**若選擇存檔：**

- [ ] 根據需求描述自動生成 slug（英文關鍵詞、`-` 連接，例如 `ad-performance-dashboard`）
- [ ] 若專案根目錄的 `docs/ideas/` 不存在，先執行 `mkdir -p docs/ideas`
- [ ] 以 Write tool 將訪談內容寫入專案根目錄的 `docs/ideas/{slug}.md`；若檔案已存在，先詢問使用者要**覆蓋**還是**加後綴**（例如 `{slug}-2.md`），不要預設覆蓋
- [ ] 提示使用者將 `docs/ideas/` 加入 `.gitignore`（若尚未加入）
- [ ] 提示下一步：

> **接下來你可以** 將這份文件分享給 PM 或利害關係人確認，或交給 PRD 工具繼續展開規格。

**存檔格式原則：**

結構可依訪談實際內容彈性調整，但目標是資訊密度足夠，讓接收這份文件的 PRD 工具不需要重新訪談。

**最小必含欄位**（缺一不可，即使內容只有「訪談中未觸及」或「TBD」也要列出來，以提醒 PRD 工具這裡是空缺）：

- **背景與動機**：為什麼要做、現況問題是什麼、如果不做最壞的情況
- **成功指標**：至少一個可量測的判斷依據（數字、頻率、時間等），避免空泛描述
- **核心決策記錄**：每個重要決策都要附「為什麼」（有哪些選項、為何採用這個）
- **未解決問題**：需後續確認的問題，每題標記負責人（TBD 或具名）

**選用欄位**（視訪談實際觸及內容加入，沒聊到就省略）：

- 使用者 / 系統角色與情境：誰在用、在什麼情境下用
- 資料範疇與邊界
- Out of Scope：明確排除的功能或情境
- 影響架構的產品決策：任何牽動資料模型或系統邊界的選擇

**若不存檔：**

提示使用者可手動複製訪談摘要，或隨時再次觸發此 skill 繼續存檔。結束前仍應口頭確認下一步：
> 「好，暫時不存檔。你接下來打算怎麼做？」

## 常見藉口與反駁

Agent 傾向於以下理由跳過步驟，但這些理由都不成立：

| 藉口 | 反駁 |
|------|------|
| 「需求已經夠清楚了，可以直接寫 PRD。」 | 「夠清楚」是主觀感覺，訪談才能暴露隱藏假設。先問再寫，成本遠低於寫錯重來。 |
| 「使用者說得很詳細，不需要再問了。」 | 詳細不等於完整。決策樹的每條分支都要有答案，不是問題數量夠多就算收斂。 |
| 「我可以一次問多個問題，節省來回次數。」 | 多問同時出現會讓使用者選擇性略過難題。一次一題才能逼出真正的思考。 |
| 「使用者說不知道，就跳過這個問題吧。」 | 不知道是探索的起點，不是終點。先給範例、再試反面切入，最後才記為未解決。 |
| 「訪談看起來差不多了，可以結束。」 | 「差不多」不是收斂標準。判斷依據是：資訊是否足以草擬 PRD，且雙方理解已一致。 |
| 「產品決策問完了，架構的事之後再說。」 | 有些決策表面是產品選擇，但早一步確認可以省去大幅重構。訪談是最便宜的時機。 |

## 警示紅旗

出現以下跡象時，應立即暫停並重新評估：

- 使用者的回答一直是「都可以」、「看情況」→ 表示問題切入角度太抽象，需換具體情境範例
- 某條子分支始終沒有縮小（連續幾個回合都在同一個問題打轉）→ 可能是訪談方向偏離核心問題
- 使用者對某個問題表現出困惑或不耐 → 重新檢查選項設計是否互斥且貼近實際情境
- 訪談過程中出現新的、更根本的問題（例如：「其實我們連要不要做都還沒確定」） → 立刻切換到這個新問題，重新建立決策樹，不要繼續在舊問題上挖下去ㄋ

## 驗證要求

「感覺問得差不多了」不是完成的依據。完成此技能須確認以下所有項目：

- [ ] 需求的「為什麼」和「成功標準」已清楚，足以讓工程師評估優先順序
- [ ] 至少一個成功指標是可量測的（數字、頻率、時間等），非空泛描述
- [ ] 核心功能邊界明確，Out of Scope 有明確共識
- [ ] 每個核心功能區域至少有一個具體 User Story（角色 + 觸發情境 + 預期結果）
- [ ] 技術邊界草稿存在：大致知道要新建或修改哪些模組，或明確記為未解決問題
- [ ] 重要的架構影響點（產品×架構交界層）已觸及或記錄為待確認
- [ ] 已完成隱含依賴掃描，確認核心功能的配套需求都已涵蓋
- [ ] 使用者已明確確認可以收斂（口頭確認或選擇存檔）
- [ ] 若選擇存檔：專案根目錄的 `docs/ideas/{slug}.md` 已建立，資訊密度足夠供 PRD 工具使用，不需重新訪談
- [ ] 使用者已收到下一步提示

## 訪談規則速查

| 規則 | 說明 |
|------|------|
| 每回合一題 | 不要一次丟多個問題，讓使用者逐一聚焦 |
| 問題類型決定格式 | 決策型問題給建議；現況確認型問題只列選項，不給建議 |
| 繁體中文輸出 | 所有訪談內容與文件一律用繁體中文 |
| 不主動探 codebase | 若使用者無法回答且問題可透過 Codebase 確認，用另一個 Agent 處理；否則不主動翻閱程式碼 |
| 只做訪談與文件 | PRD、User Stories、AC、task 拆解、code 全部交由後續工具處理；這個 skill 的邊界在訪談與存檔 |
