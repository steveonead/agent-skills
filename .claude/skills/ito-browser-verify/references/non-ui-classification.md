# 驗收標準分類規則

本檔供 `ito-browser-verify` 主流程於步驟 3 擷取「UI 可驗證」與「非 UI」兩類標準的分類原則，作為資料參照，非執行指令。

## 核心判別問題

針對每條驗收標準，以下列問題作為分類依據：

> 此條標準描述的現象，是否能在瀏覽器畫面上透過操作與觀察直接判定為符合或不符合？

- 若答案為「是」，歸類為 UI 可驗證。
- 若答案為「否」，歸類為非 UI（跳過）。

## UI 可驗證的典型句型

- 「使用者可以 [操作]，並看到 [畫面狀態]。」
- 「點擊 [按鈕／連結] 後，頁面應導向 [URL 或 view]。」
- 「輸入 [條件] 後，[元素] 應顯示／隱藏／變色。」
- 「錯誤發生時，使用者應看到 [訊息／icon／toast]。」
- 「在 [裝置尺寸] 下，[layout] 應為 [呈現方式]。」

## 非 UI 的典型句型

- 「API `POST /x` 應回傳 [status／payload]。」
- 「資料庫中 [table] 應建立一筆對應記錄。」
- 「系統每 N 分鐘應觸發一次 [job]。」
- 「log 應記錄 [事件]。」
- 「[metric] 應上報至 [監控系統]。」
- 「[function] 的複雜度應低於 [threshold]。」

## 邊界案例

### 案例 1：UI 現象與後端現象並存

例：「使用者按下送出後，畫面顯示成功 toast，且資料庫寫入一筆記錄。」

處理方式：

- 拆分為兩條 AC：前半「顯示成功 toast」歸 UI 可驗證；後半「資料庫寫入」歸非 UI。
- 於報告中保留原始句，但註記拆分結果。

### 案例 2：UI 僅為間接證據

例：「後端 job 成功後，dashboard 的計數器應 +1。」

處理方式：

- 標記為 UI 可驗證，但於 plan 中註明「需使用者先於外部觸發 job」作為 prerequisites。
- 若使用者無法觸發，改為非 UI 跳過。

### 案例 3：需要跨 session 或跨使用者的觀察

例：「A 使用者動作後，B 使用者畫面應即時更新。」

處理方式：

- 僅當使用者能同時準備兩個 browser session 且工具支援並行操作時，標為 UI 可驗證。
- 否則標為非 UI 跳過，跳過原因記錄為「需多 session 並行，超出單一瀏覽器工具能力」。

### 案例 4：效能或時間相關

例：「搜尋結果應於 2 秒內返回。」

處理方式：

- 若僅需粗略確認「有無明顯延遲」，可標為 UI 可驗證，於報告中以主觀描述記錄。
- 若需精確量測毫秒級，超出瀏覽器操作工具的合理職責，標為非 UI 跳過。

## 跳過原因標註格式

非 UI AC 於 Planning 與報告中統一以下列格式標註：

```
⏭️ Skipped — <原因類別>：<一句話說明>
```

原因類別至少區分：

- `API 契約`
- `資料層`
- `背景工作／排程`
- `可觀測性（log／metric）`
- `非功能性（效能／安全）`
- `多 session 並行`
- `其他`
