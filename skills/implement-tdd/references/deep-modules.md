# Deep Modules

出自 *A Philosophy of Software Design*：

**Deep module** = 小介面 + 厚實作

```
┌─────────────────────┐
│   Small Interface   │  ← 少數方法、簡單參數
├─────────────────────┤
│                     │
│                     │
│  Deep Implementation│  ← 複雜邏輯被藏起來
│                     │
│                     │
└─────────────────────┘
```

**Shallow module** = 大介面 + 薄實作（要避免）

```
┌─────────────────────────────────┐
│       Large Interface           │  ← 許多方法、複雜參數
├─────────────────────────────────┤
│  Thin Implementation            │  ← 只是轉手
└─────────────────────────────────┘
```

設計介面時自問：

- 這個方法**真的**是 caller 需要的，還是洩漏內部細節？
- Caller 需要知道多少內部結構才能用？越少越好
- 如果實作整個換掉，介面會不會跟著變？會的話，介面太淺

---

## 為什麼 Deep Module 對 TDD 重要？

TDD 只透過 public interface 測試。淺模組 = 介面暴露一堆內部細節 = 測試被迫測一堆 shape 細節 = 一搬程式就壞。

**深模組讓你有最大空間改內部而不壞測試**。這是 TDD 的核心收益之一：實作可以自由演化，測試像契約一樣穩定。
