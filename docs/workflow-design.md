# 工程開發 Workflow 設計決策

> 基於 Matt Pocock 的 skills 流程 grill-me → write-a-prd → prd-to-issues → tdd （https://github.com/mattpocock/skills/tree/main）客製化，保留核心設計哲學，適配團隊工具與語言環境。

---

## 核心設計哲學（不可妥協）

1. **先想清楚再動手**：grill-me → PRD → tickets 這條鏈確保在寫第一行程式碼之前，所有決策分支都被解決
2. **Deep module 設計**：模組介面簡單、實作複雜、可獨立測試；介面穩定才能讓測試長期有效
3. **垂直切片（tracer bullets）**：每個 ticket、每個測試都是端到端的薄切片，而不是水平分層
4. **行為測試，而非實作測試**：測試驗證 public interface 的行為，重構時測試不應該壞掉

> Deep module 是 2、3、4 的橋樑：因為模組是 deep 的，才能用 public interface 寫出穩定的行為測試；垂直切片確保每個 ticket 都能獨立驗收。

---

## 流程總覽

```
squeeze-idea → idea-to-prd → prd-to-ticket → implement-tdd
```

每個 skill 結尾明確提示下一步，讓流程銜接不依賴使用者記憶。

---

## 技術環境

| 面向 | 決策 |
|------|------|
| 語言 | TypeScript |
| 單元 / 整合測試 | Vitest（`npx vitest run`） |
| E2E 測試 | Playwright（`npx playwright test`） |
| 測試策略 | 測試金字塔：Vitest 為主，Playwright 只覆蓋關鍵使用者旅程 |
| 專案管理工具 | Jira（手動建立，AI 輸出結構化內容供複製貼上） |
| 輸出語言 | 全部繁體中文台灣用語，技術名詞保留英文原文 |

## Jira 映射關係

| 原版概念 | Jira 對應 |
|----------|-----------|

| Vertical slice issue | Sub-task（掛在 Story 下） |

## 各 Skill 角色與設計細節

### `squeeze-idea`（原 `https://github.com/mattpocock/skills/blob/main/grill-me/SKILL.md`）

**角色**：流程入口。從產品思維角度把想法壓榨清楚，在進入工程規劃之前確認問題本質、使用者情境與成功定義。

**設計原則**：
- 保持泛用性，不限定工程場景（也可用於非工程討論）
- 一次只問一個問題
- 每題附上 AI 的推薦答案
- 根據想法性質，**選擇性**從以下面向追問（不需要全問，但不應全部跳過）：

| 面向 | 目的 |
|------|------|
| 問題根源 | 使用者真正的痛點是什麼？現在怎麼解決的？ |
| 使用者與情境 | 誰會用？在什麼情況下用？ |
| 成功定義 | 怎麼知道這個功能成功了？ |
| 替代方案 | 有沒有其他解法被考慮並排除？為什麼？ |
| 範疇邊界 | 什麼明確不在這次要解決的範圍內？ |

**結尾流程**：

訪談結束後詢問使用者選擇：
1. **直接繼續** → 提示執行 `/idea-to-prd`，將討論結論直接作為輸入
2. **先存檔** → 根據討論內容自動建議中文檔名，存至 `docs/ideas/`；格式由 AI 自行判斷，確保另一個 AI 實例能理解並繼續工作

**與原版差異**：更名；新增產品思維的五個探索面向（選擇性）；新增本地存檔選項；新增流程銜接提示。

### `idea-to-prd`（原 `https://github.com/mattpocock/skills/blob/main/write-a-prd/SKILL.md`）

**角色**：核心規劃 skill。透過 codebase 探索、工程導向訪談、模組設計，產出一份完整的 PRD，以 Jira Story 格式輸出供手動建立。

**流程步驟**：
1. 請使用者描述問題與初步想法（**若已執行 `squeeze-idea`，跳過此步驟，直接請使用者提供討論結論作為輸入**）
2. 探索 codebase，驗證技術可行性與現有狀態
3. 進行工程導向訪談（見下方強化說明）
4. 設計 deep modules，確認介面與測試邊界
5. 輸出 PRD（Jira Story 格式）

**Step 3 工程訪談強化**：

根據功能性質，**選擇性**從以下面向追問（不需要全問，但不應全部跳過）。問題一律從**使用者行為視角**出發，而非實作細節：

| 面向 | 目的 | 示例問法 |
|------|------|----------|
| 需求模糊地帶 | 消除多重解讀、確認邊界條件 | 「當 X 情況發生時，預期行為是什麼？」 |
| 技術整合點 | 確認與現有系統的介面邊界 | 「這個功能需要和哪些現有 API 或服務互動？」 |
| 風險與未知 | 識別不確定性與外部依賴 | 「這裡有沒有你還不確定可行性的部分？」 |
| 使用者可感知的效能預期 | 影響架構決策（從使用者視角） | 「使用者預期等待多久是可接受的？」 |
| 失敗情境的使用者體驗 | 影響 module interface 設計（從使用者視角） | 「當外部服務失敗時，使用者應該看到什麼？」 |

> 這些面向是引導方向，不是強制清單。AI 應根據功能複雜度自行判斷哪些相關。

**PRD 輸出格式（Jira Story Description）**：

```
## 問題陳述
（從使用者視角描述問題）

## 解決方案
（從使用者視角描述解決方案）

## 使用者故事
（大量、編號清單）
1. 身為 <角色>，我希望 <功能>，這樣可以 <效益>
...

## 實作決策
（模組設計、介面、API 契約、schema 變更、架構決策、具體互動）
（不包含具體 file path 或程式碼片段）

## 測試決策
（好的測試定義：只測外部行為，不測實作細節）
（哪些模組要測試）
（codebase 中類似測試的先例）

## 範疇外
（明確列出不在本次範圍內的項目）

## 補充說明
（其他備註）
```

**結尾提示**：「PRD 已完成，建議執行 `/prd-to-ticket`」

**與原版差異**：Step 1 可跳過（若已執行 `squeeze-idea`）；Step 3 加入五個工程訪談面向；PRD 模板章節改為中文；輸出格式從 `gh issue create` 改為 Jira Story 結構化文字。

### `prd-to-ticket`（原 `https://github.com/mattpocock/skills/blob/main/prd-to-issues/SKILL.md`）

**角色**：將 PRD 拆解為可獨立執行的 Jira Sub-tasks，使用垂直切片原則，每個 Sub-task 是端到端的薄切片。

**流程步驟**：
1. 取得 PRD 內容（使用者提供 Jira Story 連結或直接貼入）
2. 若尚未探索 codebase，進行探索
3. 草擬垂直切片，每個切片分類為 HITL 或 AFK
4. 向使用者確認切片粒度、依賴關係、HITL/AFK 標記
5. 輸出所有 Sub-tasks 的結構化內容（供手動建立）

**垂直切片規則**：
- 每個切片切穿所有層（schema、API、UI、tests）
- 完成的切片可獨立 demo 或驗證
- 優先多個薄切片，而非少數厚切片
- AFK 優先於 HITL（盡量讓 AI 可以直接實作，不需人工介入）

**Sub-task 輸出格式**：

```
## 父 Story
（Jira Story 標題）

## 要建構的內容
（描述這個垂直切片的端到端行為，不分層描述；引用 PRD 相關段落而非重複內容）

## 驗收標準
- [ ] 標準一
- [ ] 標準二
- [ ] 標準三

## 被阻擋於
（被阻擋於：<Sub-task 標題>）
（或：無，可立即開始）

## 對應的使用者故事
（引用 PRD 中的編號）
- 使用者故事 3
- 使用者故事 7
```

**結尾提示**：「Tickets 已產出，可開始執行 `/implement-tdd`」

**與原版差異**：輸出從 `gh issue create` 改為結構化文字供手動建立；引用方式從 `#number` 改為標題；欄位標題中文化。

### `implement-tdd`（原 `https://github.com/mattpocock/skills/blob/main/tdd/SKILL.md`）

**角色**：執行階段。以紅綠重構循環逐一實作 Sub-task，每個循環只處理一個行為。

**測試哲學**：
- 測試驗證行為，不驗證實作
- 只使用 public interface 測試
- 重構時測試不應該壞掉
- 測試讀起來像規格書

**測試金字塔**：
- **底層（多）**：Vitest 單元測試，覆蓋商業邏輯、資料轉換、deep module 介面
- **中層**：Vitest 整合測試，覆蓋跨模組互動、API 層
- **頂層（少）**：Playwright E2E，只覆蓋關鍵使用者旅程的 happy path

**測試指令**：
- 單元 / 整合：`npx vitest run`
- E2E：`npx playwright test`

**循環流程**：
```
規劃 → 確認介面與測試範圍
  ↓
RED   → 寫一個測試，確認失敗
  ↓
GREEN → 寫最少程式碼讓測試通過
  ↓
重複  → 下一個行為
  ↓
重構  → 所有測試通過後才重構
```

**每個循環的檢查清單**：
- [ ] 測試描述行為，不描述實作
- [ ] 測試只使用 public interface
- [ ] 測試可以撐過內部重構
- [ ] 程式碼是通過本次測試的最少量
- [ ] 沒有加入投機性功能

**反模式（嚴格禁止）**：
- 水平切片：一次寫完所有測試再寫所有實作
- 測試 private method
- Mock 內部 collaborator（除非是外部系統邊界）
- 在 RED 狀態下重構

**與原版差異**：新增 Vitest + Playwright 的具體指令與使用時機；明確說明測試金字塔分層；新增「流程終點」標記。

## 與原版的核心差異總結

| 面向 | 原版 | 客製版 |
|------|------|--------|
| 流程銜接 | 各 skill 獨立 | 每個 skill 結尾提示下一步 |
| 想法探索框架 | 無（純泛用訪談） | 五個產品思維面向的選擇性引導 |
| 想法存檔 | 無 | 可存至 `docs/ideas/`（中文檔名，格式由 AI 自行決定） |
| idea-to-prd 入口 | 固定從 step 1 開始 | 若已執行 squeeze-idea 可跳過 step 1 |
| 專案管理 | GitHub Issues（`gh` CLI 自動建立） | Jira（AI 輸出結構化內容，手動建立） |
| 語言 | 英文 | 繁體中文台灣用語 |
| 工程訪談 | 泛用「逐一解決決策分支」 | 五個工程面向的選擇性引導（使用者行為視角） |
| 測試框架 | 框架無關 | Vitest + Playwright，測試金字塔 |
| 票券映射 | PRD → issue，切片 → issue | PRD → Story，切片 → Sub-task |
